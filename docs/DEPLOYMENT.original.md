# Deployment

Deployment guide for [Agent Web](../README.md), a self-hosted AI terminal agent organized as a pnpm monorepo. The application deploys exclusively through Docker — there is no standalone bare-metal or serverless deployment path.

## Build Pipeline

### Production build command

```bash
pnpm docker:prod
```

Runs `docker compose -f docker-compose.prod.yml up --build -d`, which:

1. Builds the `production` target of `Dockerfile` (multi-stage: `base` → `deps` → `builder` → `production`).
2. Builds the `sandbox` target in parallel.
3. Starts both containers detached.

### Multi-stage Dockerfile breakdown

| Stage | Base Image | What Happens |
|---|---|---|
| `base` | `node:22-slim` | Enables Corepack, activates pnpm 9.0.0 |
| `deps` | `base` | Installs system build deps (python3, make, g++, ca-certificates, curl), copies `package.json` files for all workspaces, runs `pnpm install` |
| `builder` | `deps` | Copies all source, runs `pnpm build` (Turborepo orchestrates the dependency-ordered build) |
| `production` | `node:22-slim` | Copies `.next/standalone` output, `.next/static`, and `public/` from builder; installs Docker CE CLI for sandbox Docker-in-Docker; creates `packages/db/data/` dir; sets `NODE_ENV=production` |
| `development` | `deps` | Copies source, builds workspace packages (`@agent-web/core`, `@agent-web/db`), sets `NODE_ENV=development`, runs dev entrypoint script |
| `sandbox` | `node:22-slim` | Installs Python 3 + pip + git + jq + tsx + common Python packages; creates non-root `sandbox-user`; idles until invoked |

### Build output

The `production` target runs `node apps/web/server.js` — the Next.js standalone server generated by `output: "standalone"` in `apps/web/next.config.ts`. This server bundles all required dependencies and can run without additional install steps.

## Environment Setup (Production)

Copy `.env.example` (from the project root) to a location accessible to the Docker container and supply the required variables. For production deployment, the following are **required**:

| Variable | Required | Description |
|---|---|---|
| `OPENAI_API_KEY` | Yes \* | OpenAI or OpenRouter API key for LLM access |
| `OPENROUTER_API_KEY` | Yes \* | Alternative if using OpenRouter |
| `ENCRYPTION_KEY` | **Yes** | 32-byte hex key for encrypting stored API keys. Generate with `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`. Without this, a dev-only fallback is used — **insecure for production**. |

| `DATABASE_AUTH_TOKEN` | — | Required only when using remote Turso/libsql. |
| `NODE_ENV` | — | Set to `production` by the Compose file. |

\* At least one LLM provider key is required.

Set these via the `environment` block in `docker-compose.prod.yml`, an `.env` file loaded by Docker Compose (`env_file:`), or your orchestration platform's secret manager. <!-- VERIFY: Production secret injection method depends on the orchestrator (Docker Swarm secrets, Kubernetes Secrets, or manual `.env` file). -->

## Database Setup

### Local SQLite (default)

The production Compose file mounts a named volume for persistence:

```yaml
volumes:
  - db_data:/app/packages/db/data
```

The database file is created automatically on first startup by `ensureMigrated()` in `@agent-web/db`. No manual migration step is required.

### Remote Turso / libsql

To use a remote database:

1. Uncomment the Turso URL and auth token in your environment configuration:
   ```
   DATABASE_URL=libsql://your-db-url.turso.io
   DATABASE_AUTH_TOKEN=your-auth-token
   ```
2. Remove or comment out the `db_data` volume mount — it is not needed for remote databases.
3. The `@libsql/client` package handles the network connection automatically.

<!-- VERIFY: Turso database URL and auth token format are provider-specific. Ensure the Turso instance is created before deploying. -->

### Database reset

Delete the `db_data` Docker volume and restart:

```bash
docker compose -f docker-compose.prod.yml down -v
docker compose -f docker-compose.prod.yml up -d
```

Migrations are idempotent and will recreate the schema on next startup.

## Production Docker Compose Configuration

**File:** `docker-compose.prod.yml`

### `app` service

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: production
    container_name: agent-web
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=file:./packages/db/data/local.db
      - NODE_ENV=production
      - HOSTNAME=0.0.0.0
      - PORT=3000
    volumes:
      - db_data:/app/packages/db/data
      - /var/run/docker.sock:/var/run/docker.sock:ro
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: "4"
          memory: 4G
        reservations:
          cpus: "1"
          memory: 512M
```

Key aspects:

- **Port**: 3000 (exposed to host).
- **Docker socket**: Mounted read-only (`ro`) to enable the app to spawn sandbox containers via Docker-in-Docker.
- **Database volume**: Named volume `db_data` for SQLite persistence outside the container lifecycle.
- **Resource limits**: 4 CPUs / 4 GB memory limit with 1 CPU / 512 MB reservation.
- **Restart policy**: `unless-stopped`.

### `sandbox` service

```yaml
  sandbox:
    build:
      context: .
      dockerfile: Dockerfile
      target: sandbox
    container_name: agent-web-sandbox
    volumes:
      - sandbox_workspace:/workspace
    environment:
      - SANDBOX_ENABLED=true
      - SANDBOX_MAX_CPU_MS=5000
      - SANDBOX_MAX_MEMORY_MB=256
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: "2"
          memory: 1G
```

- Runs as non-root `sandbox-user`.
- Pre-installed with Python 3, pip, git, jq, tsx.
- Resource-limited to 2 CPUs / 1 GB.
- Used for isolated code execution (when `TERMINAL_BACKEND=docker`).

### Volumes

```yaml

  db_data:
  sandbox_workspace:
```

## Standalone Next.js Output

The Next.js application is configured for standalone deployment in `apps/web/next.config.ts`:

```typescript
const nextConfig: NextConfig = {
  output: "standalone",
  transpilePackages: ["@agent-web/core", "@agent-web/db"],
  serverExternalPackages: ["@libsql/client", "pdf-parse", "mammoth", "xlsx"],
};
```

- **`output: "standalone"`**: Produces a self-contained server in `.next/standalone/` that includes all necessary `node_modules`. The Docker production stage copies only this directory (plus static assets and public files).
- **`transpilePackages`**: Forces Next.js to transpile the two internal workspace packages (`@agent-web/core`, `@agent-web/db`).
- **`serverExternalPackages`**: Marks packages with native modules or large parsers as external to the Webpack bundle.

To inspect the standalone server locally:

```bash
cd apps/web && pnpm build
ls .next/standalone/
```

## Sandbox Container (Code Execution)

The sandbox container provides an isolated environment for running untrusted code. It is started automatically by `docker compose -f docker-compose.prod.yml up --build -d` (no separate profile in production).

### Capabilities

- **Node.js** execution via `tsx` (pre-installed globally)
- **Python 3** with `requests`, `beautifulsoup4`, `pandas`, `numpy`
- **Shell commands** (git, jq, curl, standard Unix utils)

### Resource constraints

| Metric | Limit |
|---|---|
| CPU | 2 cores |
| Memory | 1 GB |
| Execution timeout | 5,000 ms (configurable via `SANDBOX_MAX_CPU_MS`) |
| Max memory per execution | 256 MB (configurable via `SANDBOX_MAX_MEMORY_MB`) |

### Security

- Runs as `sandbox-user` (non-root) with restricted home and temp directories.
- Commands are dispatched via `docker exec` from the main app container (requires the Docker socket mount).
- The app-side code blocks dangerous Node.js patterns (`child_process`, `fs`, `net`, `eval`, `Function()`, `process.exit()`).

## Production Considerations

### Rate Limiting

The Next.js middleware (`apps/web/middleware.ts`) implements an **in-memory sliding-window rate limiter** for all `/api/*` routes:

- **Limit**: 60 requests per 60-second window per IP+path combination.
- **Response**: HTTP `429` with `Retry-After` and `X-RateLimit-*` headers on exhaustion.
- **Limitation**: In-memory only — resets on container restart. For multi-replica or production-critical deployments, consider replacing with an external rate limiter (e.g., Redis-based). <!-- VERIFY: Multi-instance deployments require a shared rate-limit store. -->

### Authentication

- **Session-based auth**: The middleware checks for a `session_token` cookie on protected page routes. Missing cookies redirect to `/login`.
- **Public routes**: `/login` and `/api/auth` (login, register) are always accessible.
- **API auth**: Mutating API endpoints validate the session token against the `auth_tokens` database table.
- **Static assets**: `_next/static`, `_next/image`, `favicon.ico` are unrestricted.

### API Keys

- **Server-side keys**: The chat API reads provider API keys from environment variables (`OPENAI_API_KEY`, `OPENROUTER_API_KEY`, `DEEPSEEK_API_KEY`).
- **Per-user keys**: Users can store per-provider API keys in the database via `/api/keys` endpoints.
- **Encryption at rest**: API keys in the database are encrypted using the `ENCRYPTION_KEY` environment variable. If `ENCRYPTION_KEY` is not set, a development-only fallback is used — **do not deploy to production without setting this value**.

### Security Headers and TLS

The project does not include built-in TLS termination or security headers. Deploy behind a reverse proxy (nginx, Caddy, Traefik) for:

- TLS/SSL termination <!-- VERIFY: Exact proxy setup is deployment-specific. -->
- HTTP security headers (HSTS, CSP, X-Frame-Options)
- Connection buffering and timeout management

### Monitoring

No monitoring libraries (Sentry, Datadog, New Relic, OpenTelemetry) were detected in the project dependencies. Consider adding application performance monitoring before production deployment. <!-- VERIFY: Monitoring tooling and dashboard URLs are deployment-specific. -->

## Rollback Procedure

No automated rollback pipeline is configured. To revert a production deployment:

### Option 1: Previous Docker image (recommended)

```bash
# Rebuild from a specific git tag or commit
git checkout <previous-stable-tag>
pnpm docker:prod

# Or, if the previous image is still in local cache:
docker compose -f docker-compose.prod.yml down
docker compose -f docker-compose.prod.yml up -d  # uses cached layers
```

### Option 2: Database rollback

```bash
# Stop containers
docker compose -f docker-compose.prod.yml down

# Restore from database backup
# (replace with your backup restoration command)
# docker volume inspect agent-web_db_data  # find the volume path

# Restart
docker compose -f docker-compose.prod.yml up -d
```

<!-- VERIFY: Database backup/restore procedure depends on the backup strategy chosen by the operator. -->

### Before rolling back

1. Stop incoming traffic (reverse proxy level).
2. Take a database snapshot (copy the `db_data` volume contents).
3. Deploy the previous version.
4. Run health checks before re-enabling traffic.
